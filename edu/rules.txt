Опис бізнес-правил для «Електронного журналу» (ERP/1: Освіта)
відповідно до Політик Технічних вимог, ТЗ та ТД (розділи 5.1, 3.2.1, 6.1).
1. Загальний контекст процесу
Електронний журнал — центральна реєстрова сутність, що фіксує відвідування, поточні та підсумкові оцінки, домашні завдання, зауваження та компетенції учнів/студентів за навчальними курсами.
Процес ізоморфний FSM + BPMN:

Сутності: Журнал (Record), Учень (Student), Педагог (Teacher), Курс (Course), Тема (Topic), Оцінка (Grade), Відвідування (Attendance).
Стани: Чернетка → Активний → Підписаний (КЕП) → Архівний.
Учасники: Педагог (основний), Завуч/Адміністратор (контроль), Учень/Батьки (перегляд).

2. Основні бізнес-правила (явно визначені, не імпліцитні)
2.1. Правила створення та доступу до журналу

Журнал створюється автоматично при затвердженні розкладу або вручну адміністратором для конкретного курсу/групи/семестру.
Доступ:
Педагог — повний доступ (заповнення) тільки до своїх курсів/груп (RBAC + ABAC за посадою та розкладом).
Завуч/Директор — read/write на всі журнали закладу.
Учень — read-only на свої записи.
Батьки — read-only (з обмеженням на певні поля, наприклад, без коментарів педагога).

Правило: Доступ перевіряється на кожному переході стану (включаючи API-виклики).

2.2. Правила фіксації відвідувань

Відвідування фіксується за датою та уроком (обов’язкові поля: дата, учень, статус: Присутній/Відсутній/Поважна/Неповажна).
Автоматична фіксація через QR/mobilе-check-in (з валідацією геолокації за потреби).
Правило: Після закінчення уроку (таймер) журнал переходить у режим «Підтвердження» — педагог не може змінювати відвідування без обґрунтування (коментар + підпис КЕП для змін).
Негативний сценарій: Масова відсутність (>30%) → автоматичне сповіщення завучу.

2.3. Правила виставлення оцінок та компетенцій

Оцінки — за шкалою, визначеною в навчальному плані (наприклад, 1-12, зараховано/незараховано, ECTS).
Обов’язкова валідація:
Оцінка в межах дозволеної шкали.
Не більше N оцінок за тему (конфігурується).
Середній бал розраховується автоматично (вагові коефіцієнти за типом роботи: контрольна, домашнє, поточна).

Компетенції (KPI) — мапуються на державні стандарти освіти.
Правило: Підсумкова оцінка за семестр блокується після підписання журналу КЕП педагогом.

2.4. Правила підписання та затвердження

Журнал підписується КЕП педагога після завершення періоду (тиждень/місяць/семестр).
Після підпису зміни можливі тільки через «Виправлення» з обов’язковим коментарем та повторним підписом (історія зберігається).
Завуч/Директор може затвердити (другий підпис) — після цього журнал переходить у статус «Архівний» для звітності.

2.5. Правила інтеграції та звітності

Дані журналу автоматично передаються в модулі «Аналітика», «Звітність для МОН», «Контингент».
Синхронізація з ЄДЕБО via MQTT/PKI.
Правило: Усі зміни логуються (рівень INFO/WARN для змін оцінок, AUDIT для підписів КЕП).

2.6. Обробка помилок та винятків (use case негативні сценарії)

Неможливість виставити оцінку нижче мінімуму — сповіщення + пояснення.
Конфлікт розкладу — блокування дублювання записів.
Відсутність КЕП — перехід у «Чернетка» з таймером нагадування.
Масові зміни — вимагає підтвердження завуча.

3. Вимоги до реалізації (згідно політики)

Бізнес-правила визначені явно в ТЗ (не в коді розробником).
Валідації — централізовані (окрема сутність Validations).
API: Кожен метод журналу описується за шаблоном (призначення, параметри, відповідь, помилки).
UI: Чіткі повідомлення про помилки, можливість повтору операції.
Тестування: Сценарії + протоколи (ручне + автоматичне), UAT з реальними ролями.
Критерії приймання: Журнал підписаний, дані коректно відображаються в звітах, історія повна, доступ відповідно ролям.
DoD: Реалізовано + протестовано + задокументовано + перевірено безпеку + пройдено UAT.

Ці бізнес-правила забезпечують простежуваність, аудит та відповідність Закону «Про освіту» і вимогам МОН.
Потрібен детальний BPMN-опис одного з підпроцесів журналу, Use Case у форматі таблиці, або інтеграція з іншими модулями (наприклад, «Атестація»)?
