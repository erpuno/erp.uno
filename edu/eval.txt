Деталізація BPMN підпроцесу «Оцінювання» (Електронний журнал, ERP/1: Освіта)
відповідно до Політик (розділи 5.1 Вимоги до процесів, 3.2.1 Правила до ТЗ, 6.1 Архітектура).
1. Контекст підпроцесу
Підпроцес «Оцінювання» є частиною основного бізнес-процесу «Проведення заняття та фіксація результатів» та «Підсумкове оцінювання за період».

Тригер: Завершення теми/уроку/контрольної роботи або кінець періоду (тиждень/місяць/семестр).
Сутності: Оцінка (Grade), Компетенція, Тема, Учень, Курс, Журнал.
Ролі (Lanes): Педагог (основний виконавець), Учень (учасник), Завуч (контроль), Система (автоматика).
Стани (FSM): Чернетка → Зафіксовано → Підписано (КЕП) → Підтверджено → Архів.
Вихід: Оновлення журналу, розрахунок середнього балу, сповіщення, передача в аналітику.

Детальна структура з гейтвеями та альтернативними сценаріями

Start Event — Message (завершення уроку) або Timer (підсумок періоду).
Lane: Педагог
Task: Заповнити оцінки (таблиця: учень, тип роботи, бал, коментар).

Exclusive Gateway — «Всі оцінки введено?»
No → Loop back + сповіщення педагогу.

Service Task (Система):
Валідація (див. бізнес-правила нижче).
Розрахунок середнього балу та компетенцій.

Parallel Gateway:
Гілка 1: Запис до Журналу.
Гілка 2: Сповіщення учня/батьків (push/email).
Гілка 3: Передача даних в модуль Аналітика.

User Task: Підписання КЕП (Педагог).
Exclusive Gateway: Потребує перевірки завуча?
Yes (критичні оцінки, спірні випадки) → Lane Завуч: Перегляд та підтвердження.
No → Продовжити.

End Event — Оцінка підписана, журнал оновлено.

Негативні сценарії (додаткові гілки):

Помилка валідації → Повідомлення + можливість редагування.
Відсутність КЕП → Перехід у «Очікує підпису» + нагадування (таймер 48 год).
Масова зміна оцінок → Вимагає додаткового підтвердження завуча + audit-log.

3. Бізнес-правила підпроцесу «Оцінювання» (явні)

Оцінка повинна відповідати шкалі, визначеній у навчальному плані курсу (конфігурується в довіднику).
Мінімальна/максимальна кількість оцінок за тему (наприклад, не менше 3 поточних).
Автоматичний розрахунок підсумкової оцінки за формулою (ваги: поточні 60%, контрольні 40% тощо).
Заборона зміни оцінки після підпису КЕП (крім процедури «Виправлення» з коментарем).
Учень не може отримати оцінку нижче мінімуму без фіксації додаткових заходів (репетиторство тощо).
При виставленні «незараховано» — автоматичне створення задачі на повторне проходження.
Усі дії фіксуються в історії з посиланням на КЕП підписувача.
Інтеграція: дані оцінки передаються в «Контингент», «Атестацію», «Звітність МОН».

4. Вимоги до реалізації та тестування

API-методи: POST /journal/grade, POST /journal/sign, GET /journal/student/{id} з описом параметрів, відповідей та помилок за шаблоном політики 5.2.
UI: Інформативні повідомлення про помилки, підтримка WCAG AA, mobile-first.
Критерії приймання:
Оцінка зберігається тільки після валідації.
Підпис КЕП блокує подальші зміни (крім аудитованих).
Середній бал розраховується коректно та відображається в звітах.

Definition of Done: Реалізовано + unit/integration тести + ручне тестування сценаріїв + UAT з педагогами + документація BPMN + перевірка безпеки.

Цей підпроцес повністю інтегрується з модулями «Розклад», «Контингент», «Аналітика» та «Процеси» (BPMN-двигун).
