Деталізація ER-діаграми (Entity-Relationship) для ERP/1: Освіта
відповідно до Політик (розділи 5.1, 6.1.2 — рівень «Сутності / Resources»)
1. Загальна структура ER-моделі
Основні сутності та ключові зв’язки (1:1, 1:N, N:M):

Person (базова сутність для Учень, Педагог, Батько)
Student → Group (N:1)
Teacher → Department (N:1)
Course ↔ Group (N:M через Curriculum)
JournalRecord → Student (N:1) + Course (N:1) + Teacher (N:1)
Schedule → Group/Course/Teacher/Room (N:1)
Document (поліморфний) пов’язаний з будь-якою сутністю

4. Додаткові вимоги до ER-моделі (згідно політики)

Ідентифікація: UUID + external_id (державні реєстри).
Історія змін: version + audit log (ProcessLog).
Soft Delete: deleted_at.
Валідації: На рівні бази (constraints) + application level.
Інтеграція: Синхронізація з ЄДЕБО через зовнішні ключі.
Безпека: ABAC на рівні рядків (Row Level Security) + КЕП для підписів.
Продуктивність: Індекси на FK, lesson_date, group_id тощо.

5. Рекомендації реалізації

База даних: PostgreSQL (JSONB для гнучких полів) або Mnesia/Erlang-native stores.
ORM: Ecto (Elixir) або аналог для Erlang/OTP.
Міграції: Версіоновані, з урахуванням сумісності.
Документування: Повна ER-діаграма в інструменті (dbdiagram.io / Draw.io) + експорт у ТЗ.

Ця ER-модель забезпечує простежуваність від Політик → ТВ → ТЗ → реалізація.
