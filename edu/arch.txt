Архітектура проекту ERP/1: Освіта (відповідно до Політик Технічних вимог, ТЗ та ТД 2026 © Інформаційні Судові Системи, розділ 6.1 Вимоги до архітектури).
6.1.1. Визначення термінів і призначення документа
Архітектура — формальний структурний опис компонентів системи та регламентів їх взаємодії з розширеною деталізацією прикладного рівня (згідно ISO/IEC/IEEE 42010 та фреймворку Захмана).
Призначення документа:

Ознайомити виконавців з поточною архітектурою ERP/1 (екосистема SYNRC + модуль Освіта).
Визначити технології, що використовуються.
Визначити вимоги та обмеження реалізації нової функціональності.

Проект ERP/1: Освіта — сучасна LMS/EMS-платформа для управління освітнім процесом (очна, дистанційна, змішана форми) на базі Erlang/OTP. Повністю відповідає Закону України «Про освіту», інтегрується з ЄДЕБО, державними реєстрами та екосистемою SYNRC.
6.1.2. Рівні архітектури компонентів системи
Архітектура побудована за методологією ISO/IEC/IEEE 42010 та фреймворком Захмана. Компоненти повністю визначені та узгоджені.
Основні рівні модифікацій для нової функціональності:

Сутності (Resources, ER-діаграми): Контингент (учні/студенти, особові справи), Кадри (педагоги, штатний розпис), Навчальні курси, Документи, Реєстри.
Форми (Forms): UI-форми для журналу, розкладу, атестації, KPI.
Валідації: Перевірки даних контингенту, оцінок, КЕП/X.509.
Процеси (BPMN): Навчальний процес, атестація, звітність для МОН (інтеграція з BPMN-модулем).
Довідники: Навчальні плани, предмети, компетенції.
Права (Scopes, RBAC/ABAC): Рольовий доступ (адміністратор, педагог, учень, батьки тощо).
Функції сервісів (API, RPC): MQTT/PKI інтеграції, SCORM, відеоконференції.
SQL-таблиці / KV / Cache: Erlang Mnesia / ETS / ETS-based stores + зовнішні (за потреби).
Rate limiters: Через KONG або вбудовані в Erlang/OTP.
Інтеграції: MQTT, PKI (КЕП), ЄДЕБО, «Документи», «Реєстри», «Бухгалтерія і Кадри».

Технологічний стек (агностичний опис з урахуванням існуючого):

Core: Erlang/OTP (висока масштабованість, concurrency, fault-tolerance).
Frontend: Сучасний веб-інтерфейс (відповідність WCAG AA, дизайн-код SYNRC/ERP).
Інтеграції: MQTT, PKI/X.509, SCORM, REST/gRPC.
Безпека: Вбудована криптографія X.509, КЕП, відповідність КСЗІ, GDPR, ISO/IEC.
Логування / Моніторинг: ELK-стек (відповідно до політики логування).
Масштабування: Горизонтальне (containers/orchestration за потреби), черги для важких операцій.

6.1.3. Маніфест відповідності архітектурі ІСС (адаптовано до ERP/1)

Стандартизація функцій ПЗ — за рахунок Erlang/OTP + SYNRC (єдина технологія проектування).
Відповідність міжнародним/національним стандартам (ДСТУ, ISO, NIST, ITU-T, КСЗІ).
Використання наявних систем Замовника (SYNRC екосистема: CA, Chat, LDAP, BPMN тощо).
Версійна сумісність.
Нові компоненти:
Open Source / безкоштовні залежності.
Підтримка розробника ≥1 рік.

Горизонтальне масштабування, шардинг, балансування, черги.
Мінімізація трафіку, телеметрія (операції/сек, latency, помилки).
Відповідність вимогам потужності, ємності, логування, тестування (вкл. з вимкненими КЕП).
Мінімізація вартості володіння.

Таксономія компонентів (адаптовано)

Судові/освітні компоненти (за аналогією таксономії ЄСІКС): Процеси документообігу (журнали, звіти), FSM/BPMN для навчальних процесів.
Системні компоненти: Core Erlang services, UI layer, Integration layer (MQTT/PKI), Analytics layer.

Додаткові нефункціональні аспекти (згідно політики)

Безпека: Загальні + технічні вимоги (X.509, КЕП, RBAC, логування подій).
Продуктивність: Тисячі одночасних користувачів завдяки Erlang/OTP.
UI/UX: WCAG AA, people-centric, use cases + BPMN, DoD, критерії приймання.
Тестування: Ручне + автоматичне, UAT.
Документування: Відповідно ДСТУ 3973-2000, 3008:2015 тощо.

Архітектура залишається агностичною до конкретних продуктів до затвердження ТЗ. Всі рішення — виключно на основі Політик. ТЗ повинно містити деталізацію (BPMN, реквізити сутностей, ABAC тощо).
Ця архітектура забезпечує повний цикл управління освітнім процесом з високою надійністю, безпекою та інтеграцією в державну екосистему. Для подальшої деталізації (ER-діаграми, BPMN, API-специфікації) необхідне ТЗ.
