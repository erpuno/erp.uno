Деталізація BPMN для освітніх процесів у проекті ERP/1: Освіта (відповідно до Політик, розділів 3.2, 5.1, 6.1).
Згідно Політик, Технічне завдання повинно містити BPMN-діаграми (ISO 19510), верифіковані аналітиками. Кожен процес описується в контексті бізнес-правил, use cases (основний + альтернативні/негативні сценарії), ролей, сутностей, станів FSM, критеріїв приймання та Definition of Done.
1. Загальні принципи моделювання процесів (згідно політики 5.1 та 3.2.1)
Система ізоморфна FSM + BPMN + Мережі Петрі:

Контекст процесу: Перелік документів/сутностей.
Стани: Робочі місця користувача або сервіси.
Переходи: Функції з логікою (валідації, права, бізнес-правила).
Історія: Кроки + прикріплені документи.
Учасники: Користувачі з КЕП X.509, ЄДРПОУ, посадами, ролями (ABAC/RBAC).

Основні ролі (приклади):

Адміністратор закладу
Педагог/Викладач
Учень/Студент
Батьки/Представники
Методист/Завуч
Директор
Системний адміністратор

2. Ключові освітні процеси (BPMN-рівень)
2.1. Процес «Формування та затвердження навчального плану / розкладу» (BPMN Pool: Навчальний відділ)

Start: Запит на семестр (подія Timer + Message від МОН/ЄДЕБО).
Lane 1 — Методист:
Створення/імпорт навчального плану (сутність «Навчальний план», SCORM-курси).
Валідація (відповідність державним стандартам).

Lane 2 — Завуч/Директор:
Перегляд + затвердження (паралельні гілки погодження).
Бізнес-правила: перевірка навантаження педагогів (KPI), доступності приміщень.

Gateway: Exclusive — чи є зауваження? (Loop для коригування).
Lane 3 — Автоматичний сервіс:
Генерація розкладу (з урахуванням контингенту, кадрів).
Публікація в електронному журналі.

End: Повідомлення всім учасникам (Email/MQTT), фіксація в історії.

Сутності: Навчальний план, Розклад, Курс (SCORM), Навантаження педагога.
2.2. Процес «Проведення заняття та фіксація відвідувань/оцінок» (Core навчальний процес)

Start: Початок заняття за розкладом (Timer/Event).
Lane — Педагог:
Відкриття електронного журналу (Form з контингентом групи).
Фіксація відвідувань (Check-in via мобільний/QR).
Виставлення оцінок/компетенцій (валідація шкали).

Паралельні гілки:
Учень: доступ до матеріалів курсу, тестування.
Система: автоматичний контроль знань (онлайн-тести).

Gateway: Оцінка пройдена? → Сертифікація / Повтор.
End: Запис до історії, сповіщення батькам (при негативних сценаріях), оновлення аналітики.

Альтернативні сценарії: Відсутність інтернету (offline-режим + синхронізація), хвороба учня (академічна відпустка).
2.3. Процес «Атестація та підвищення кваліфікації педагогів»

Start: Планова атестація (Timer + Message).
Учасники: Педагог → Методист → Комісія → Директор.
Кроки:
Подання особової справи + портфоліо (документ).
Оцінка KPI (автоматичний розрахунок).
Погодження (multi-sign via КЕП).
Рішення комісії.

End: Оновлення штатного розпису, формування наказу, запис до реєстру.

2.4. Процес «Формування звітності для МОН» (інтеграційний)

Автоматичний (на основі даних контингенту, журналів, атестацій).
BPMN: Data Collection → Validation (відповідність формам МОН) → Signing (КЕП) → Transmission via ЄДЕБО/MQTT.
Моніторинг статусу доставки.

3. Архітектурні аспекти реалізації BPMN (6.1.2)

Процеси моделюються як послідовність викликів API + BPMN-двигуна (інтеграція з модулем «Процеси» SYNRC).
Сутності пов’язані через ER-модель (Контингент ↔ Курси ↔ Оцінки).
Валідації та бізнес-правила — явні, не імпліцитні в коді.
Інтерфейс: React/Vue-подібний UI з підтримкою WCAG AA, people-centric use cases.
Масштабування: Erlang/OTP actors для concurrent процесів (тисячі користувачів).

1. bpe_education_assessment.erl (Оцінювання) — з умовними переходами
erlang-module(bpe_education_assessment).
-include("bpe.hrl").
-compile(export_all).

def() ->
    #process{
        name = 'Student Assessment',
        beginEvent = 'StartAssessment',
        endEvent = 'AssessmentSigned',
        tasks = [
            #beginEvent{name='StartAssessment'},
            #userTask{name='EnterGrades', module=?MODULE},
            #serviceTask{name='ValidateGrades', module=?MODULE},
            #serviceTask{name='CalculateCompetences', module=?MODULE},
            #userTask{name='TeacherSign', module=?MODULE},
            #userTask{name='ViceCheck', module=?MODULE},
            #endEvent{name='AssessmentSigned'}
        ],
        flows = [
            #sequenceFlow{name='1', source='StartAssessment', target='EnterGrades'},
            
            #sequenceFlow{name='2', 
                         source='EnterGrades', 
                         target='ValidateGrades'},
            
            #sequenceFlow{name='3', 
                         source='ValidateGrades', 
                         target='CalculateCompetences', 
                         condition="validation_ok"},
            
            #sequenceFlow{name='4', 
                         source='ValidateGrades', 
                         target='EnterGrades', 
                         condition="validation_failed"},  % loop back
            
            #sequenceFlow{name='5', 
                         source='CalculateCompetences', 
                         target='TeacherSign'},
            
            #sequenceFlow{name='6', 
                         source='TeacherSign', 
                         target='ViceCheck', 
                         condition="requires_approval"},   % критична оцінка / спірна ситуація
            
            #sequenceFlow{name='7', 
                         source='TeacherSign', 
                         target='AssessmentSigned', 
                         condition="no_approval_needed"},
            
            #sequenceFlow{name='8', 
                         source='ViceCheck', 
                         target='AssessmentSigned', 
                         condition="vice_approved"}
        ],
        roles = [teacher, vice_rector]
    }.

action({serviceTask, 'ValidateGrades'}, Proc) ->
    % Логіка повернення умови
    case validate_grades(Proc) of
        ok -> {reply, Proc, "validation_ok"};
        _  -> {reply, Proc, "validation_failed"}
    end;

action({userTask, 'TeacherSign'}, Proc) ->
    case requires_vice_approval(Proc) of
        true -> {reply, Proc, "requires_approval"};
        false -> {reply, Proc, "no_approval_needed"}
    end;

action(_, Proc) -> {reply, Proc}.

erlang-module(bpe_education_lesson).
-include("bpe.hrl").
-compile(export_all).

def() ->
    #process{
        name = 'Lesson Execution and Assessment',
        beginEvent = 'LessonStart',
        endEvent = 'LessonClosed',
        tasks = [
            #beginEvent{name='LessonStart'},
            #userTask{name='TakeAttendance', module=?MODULE},
            #userTask{name='AssessStudents', module=?MODULE},
            #serviceTask{name='CalculateAverages', module=?MODULE},
            #userTask{name='SignJournal', module=?MODULE},
            #endEvent{name='LessonClosed'}
        ],
        flows = [
            #sequenceFlow{name='1', source='LessonStart', target='TakeAttendance'},
            #sequenceFlow{name='2', source='TakeAttendance', target='AssessStudents'},
            #sequenceFlow{name='3', source='AssessStudents', target='CalculateAverages'},
            
            #sequenceFlow{name='4', 
                         source='CalculateAverages', 
                         target='SignJournal', 
                         condition="averages_ok"},
            
            #sequenceFlow{name='5', 
                         source='CalculateAverages', 
                         target='AssessStudents', 
                         condition="averages_need_correction"},  % loop
            
            #sequenceFlow{name='6', 
                         source='SignJournal', 
                         target='LessonClosed', 
                         condition="kep_signed"}
        ],
        events = [#timeoutEvent{name='JournalDeadline', timeout=#timeout{spec={hours,48}}}]
    }.

action({serviceTask, 'CalculateAverages'}, Proc) ->
    case averages_valid(Proc) of
        true -> {reply, Proc, "averages_ok"};
        false -> {reply, Proc, "averages_need_correction"}
    end;

action({userTask, 'SignJournal'}, Proc) ->
    case has_valid_kep(Proc) of
        true -> {reply, Proc, "kep_signed"};
        false -> {reply, Proc}  % залишається в поточному стані
    end;

action(_, Proc) -> {reply, Proc}.

3. Erlang BPE реалізація (bpe_curriculum_approval.erl)
erlang-module(bpe_curriculum_approval).
-include("bpe.hrl").
-compile(export_all).

def() ->
    #process{
        name = 'Curriculum Approval',
        beginEvent = 'Start',
        endEvent = 'PlanApproved',
        tasks = [
            #beginEvent{name='Start'},
            #userTask{name='CreatePlan', module=?MODULE},           % Методист
            #userTask{name='MethodistReview', module=?MODULE},
            #userTask{name='ViceApprove', module=?MODULE},          % Завуч
            #userTask{name='DirectorApprove', module=?MODULE},      % Директор
            #serviceTask{name='GenerateSchedule', module=?MODULE},
            #endEvent{name='PlanApproved'}
        ],
        flows = [
            #sequenceFlow{name='1', source='Start', target='CreatePlan'},
            #sequenceFlow{name='2', source='CreatePlan', target='MethodistReview'},
            
            #sequenceFlow{name='3', 
                         source='MethodistReview', 
                         target='ViceApprove', 
                         condition="methodist_ok"},
            
            #sequenceFlow{name='4', 
                         source='MethodistReview', 
                         target='CreatePlan', 
                         condition="methodist_reject"},
            
            #sequenceFlow{name='5', 
                         source='ViceApprove', 
                         target='DirectorApprove', 
                         condition="vice_ok"},
            
            #sequenceFlow{name='6', 
                         source='ViceApprove', 
                         target='MethodistReview', 
                         condition="vice_reject"},
            
            #sequenceFlow{name='7', 
                         source='DirectorApprove', 
                         target='GenerateSchedule', 
                         condition="director_ok"},
            
            #sequenceFlow{name='8', source='GenerateSchedule', target='PlanApproved'}
        ],
        roles = [methodist, vice_rector, director]
    }.

%% Дії
action({userTask, 'CreatePlan'}, Proc) -> {reply, Proc};
action({userTask, 'MethodistReview'}, Proc) ->
    case validate_curriculum(Proc) of
        ok -> {reply, Proc, "methodist_ok"};
        _  -> {reply, Proc, "methodist_reject"}
    end;

action({userTask, 'ViceApprove'}, Proc) ->
    case vice_approval(Proc) of
        ok -> {reply, Proc, "vice_ok"};
        _  -> {reply, Proc, "vice_reject"}
    end;

action({userTask, 'DirectorApprove'}, Proc) ->
    case director_kep_sign(Proc) of
        ok -> {reply, Proc, "director_ok"};
        _  -> {reply, Proc}
    end;

action({serviceTask, 'GenerateSchedule'}, Proc) ->
    % Запуск генерації розкладу та сповіщень
    {reply, Proc};

action(_, Proc) -> {reply, Proc}.
4. Бізнес-правила в процесі

Перевірка відповідності кредитів і годин освітній програмі.
Обов’язковий підпис КЕП на фінальному етапі.
При відхиленні — обов’язковий коментар.
Після затвердження автоматичний запуск паралельного процесу генерації розкладу.

Як працювати з умовами в BPE

Умова (condition) — атом або string, який повертається з action/2 як третій елемент tuple {reply, Proc, Condition}.
BPE автоматично обирає гілку, яка відповідає умові.
Для складних умов можна використовувати rules у #process{} або зовнішні функції.

Ці оновлення повністю відповідають бізнес-правилам (валідація оцінок, КЕП, погодження тощо) і готові до інтеграції в ERP/1: Освіта.
